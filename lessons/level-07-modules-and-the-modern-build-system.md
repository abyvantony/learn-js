# Level 7: Modules and the modern build system

**Track:** JavaScript · **Level:** 7/12 · **Cycle:** 2 · **Difficulty:** `intermediate`

📚 **Today's lesson** — published 2026-07-08

## TL;DR

Modern JavaScript uses ES Modules (`import` / `export`) to split code into reusable files. Build tools (Vite, Webpack, esbuild) bundle these modules for the browser, transforming modern JS into something older browsers can run.

## Real-world analogies

- A module is a chapter of a book. Each chapter is self-contained, but contributes to the whole. ES Modules let you `import` chapters from other books.
- A build tool is the publisher. It takes your draft (modern JS, multiple files), formats it (bundles, minifies), and ships a polished book (one or a few files) to readers (browsers).

## Key concepts

### `ES Modules`

`export const foo = 1;` in one file, `import { foo } from './bar.js'` in another. Native to the language, no library needed.

### `CommonJS (Node.js)`

`module.exports = foo` and `const foo = require('./bar')`. The older Node.js standard. Modern Node supports both.

### `Bundlers`

Vite, Webpack, esbuild, Rollup. Take your many source files and produce one (or a few) optimized files for the browser.

### `Package managers`

npm (default), pnpm (faster, disk-efficient), yarn (Facebook). They install third-party packages from npmjs.com.

```
// === ES Modules (modern, native) ===
// math.js
export const add = (a, b) => a + b;
export const subtract = (a, b) => a - b;
export default class Calculator {  // One default export per file
    multiply(a, b) { return a * b; }
}

// app.js
import { add, subtract } from './math.js';
import Calculator from './math.js';

console.log(add(2, 3));           // 5
console.log(new Calculator().multiply(2, 3));  // 6

// === CommonJS (Node.js legacy) ===
// math.cjs
const add = (a, b) => a + b;
module.exports = { add, subtract };

// app.cjs
const { add } = require('./math.cjs');

// === package.json ===
{
    "name": "my-app",
    "version": "1.0.0",
    "type": "module",  // Use ESM instead of CommonJS
    "scripts": {
        "start": "node app.js",
        "dev": "vite",
        "build": "vite build",
        "test": "vitest"
    },
    "dependencies": {
        "react": "^18.0.0",
        "axios": "^1.6.0"
    },
    "devDependencies": {
        "vite": "^5.0.0",
        "vitest": "^1.0.0"
    }
}

// === Common npm commands ===
// npm init -y                    # Create package.json
// npm install <package>          # Add a dependency
// npm install --save-dev <pkg>   # Add a dev dependency
// npm install                    # Install everything from package.json
// npm run <script>               # Run a script
// npm update                     # Update packages
// npm outdated                   # See what's outdated

// === Import patterns ===
import defaultExport from './module.js';           // Default
import { named1, named2 } from './module.js';      // Named
import * as everything from './module.js';         // Namespace
import { named as alias } from './module.js';      // Renamed
import('./module.js').then(m => m.foo());           // Dynamic import (lazy)

// === Dynamic import for code-splitting ===
async function loadChart() {
    const { Chart } = await import('./chart-library.js');
    return new Chart();
}
```

## Try it yourself

Create a project with two files: `math.js` (exports add, subtract, multiply) and `app.js` (imports them, uses them). Add a `package.json` with `"type": "module"`. Run with `node app.js`.

## Common pitfalls

- ⚠️ Mixing CommonJS and ESM. Pick one. Use `"type": "module"` in package.json for ESM.
- ⚠️ Forgetting the file extension in imports: `import x from './foo.js'`, not `'./foo'` (in Node.js; bundlers often allow extensionless).
- ⚠️ Importing circular references (A imports B, B imports A). Causes undefined values. Restructure to avoid.
